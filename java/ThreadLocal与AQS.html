    ThreadLocal与AQS

    lin 2023/9/30 JUC

    # 一. ThreadLocal

    # 1.什么是ThreadLocal

    线程本地存储机制

    ThreadLocal类主要解决的就是让每个线程绑定自己的值，可以将ThreadLocal类形象的比喻成存放数据的盒子，盒子中可以存储每个线程的私有数据。

    # 2.操作

    # put操作

    ThreadLocalMap 为 ThreadLocal 的一个静态内部类，里面定义了Entry 来保存数据.

    在Entry内部使用ThreadLocal作为key(是弱引用)，使用我们设置的value作为value。

    ####　get操作

    首先获取当前线程，然后通过key threadlocal 获取 设置的value 。

    # 3.内存泄漏

    如果ThreadLocal是null

    会被垃圾回收器回收

    但是ThreadLocalMap的生命周期和Thread一样不会被回收

    这是,ThreadLocalMap的Key没了,但value还在就造成了内存泄漏

    解决方式:

    使用完ThreadLocal后执行remove操作

    # 二.AQS

    # 1. 什么是AQS?

    JDK提供的一个同步框架,内部维护这FIFO双向队列,AQS依赖它来完成同步状态管理

    如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态。

    如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制，这个机制 AQS 是用 CLH 队列锁 实现的，即将暂时获取不到锁的线程加入到队列中

    AQS 是将每条请求共享资源的线程封装成一个 CLH 锁队列的一个结点（Node）来实现锁的分配。

    AQS就是基于CLH队列，用volatile修饰共享变量state，线程通过CAS去改变状态符，成功则获取锁成功，失败则进入等待队列，等待被唤醒。

    # 2.AQS原理

    AQS维护了与i个 volatile int state 变量 和 一个 CLH 双向队列

    state变量:

    • 0 : 表示没有任何线程竞争锁资源
    • 1 : 表示已经有线程正在持有锁资源

    队列中的节点持有线程引用，每个节点均可通过getState()、setState()和compareAndSetState()对state进行修改和访问

    当线程获取锁时，即试图对state变量做修改，如修改成功则获取锁；如修改失败则包装为节点挂载到队列中，等待持有锁的线程释放锁并唤醒队列中的节点。

    未获取到锁的线程会被阻塞,加入到双向队列中

    # 3.AQS核心结构

    # 1. Node

    如果AQS线程获取资源失败,会包装成一个节点挂在到CLH队列上

    Node类主要包括五个字段

    • waitStatus : 当前节点状态,共有五个取值
      • 1 --- 节点引用线程由于等待超时或被打断时的状态
      • -1 --- 后继节点线程需要被唤醒时的当前节点状态,表示需要被唤醒
      • -2 --- 当节点线程进入 condition 队列时的状态
      • -3 --- 仅当释放共享锁 releaseShare时对头节点时使用
      • 0 --- 节点初始化状态
    • prev : 前驱节点
    • next : 后继节点
    • thread : 应用线程 , 头节点不包含线程
    • nextWaiter ; condition 条件队列

    # 2.独占锁分析

    # acquire

    acquire核心为tryAcquire、addWaiter和acquireQueued三个函数，其中tryAcquire需具体类实现。 每当线程调用acquire时都首先会调用tryAcquire，失败后才会挂载到队列，因此acquire实现默认为非公平锁。

    addWaiter将线程包装为独占节点，尾插式加入到队列中，如队列为空，则会添加一个空的头节点。值得注意的是addWaiter中的enq方法，通过CAS+自旋的方式处理尾节点添加冲突。

    acquireQueue在线程节点加入队列后判断是否可再次尝试获取资源，如不能获取则将其前驱节点标志为SIGNAL状态(表示其需要被unpark唤醒)后，则通过park进入阻塞状态。

    # release

    release流程较为简单，尝试释放成功后，即从头结点开始唤醒其后继节点，如后继节点被取消，则转为从尾部开始找阻塞的节点将其唤醒。阻塞节点被唤醒后，即进入acquireQueued中的for(;;)循环开始新一轮的资源竞争。

    # 3.共享锁分析

    即读锁

    acquireShared和releaseShared整体流程与独占锁类似，tryAcquireShared获取失败后以Node.SHARED挂载到队尾阻塞，直到队头节点将其唤醒。在doAcquireShared与独占锁不同的是，由于共享锁是可以被多个线程获取的，因此在首个阻塞节点被唤醒后，会通过setHeadAndPropagate传递唤醒后续的阻塞节点。
