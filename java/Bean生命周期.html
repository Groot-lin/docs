    # 创建前的准备

    在bean的生命周期开始前

    Spring容器启动后

    会先执行loadBeanDefinitions()方法得到bean定义的集合

    # Bean的生命周期

    执行启动类,会输出对应结果

    所以Bean的生命周期分别是

    1. 实例化
    2. 属性填充
    3. 初始化
    4. 销毁

    # 1.实例化

    Bean实例化基本流程 :

    1. BeanDefinnitionReader加载xml配置文件,解析获取配置中每个bean的信息,封装成一个个的BeanDefinition对象
    2. 将BeanDefinition存储在一个名为beanDefinnitionMap的Map<String ,BeanDefinition>中
    3. ApplicationContext底层遍历beanDefinitionMap,创建Bean实例对象
    4. 创建好的Bean实例对象,被存储到一个名为singletonObject(单例池)的Map<String ,Object>中
    5. 当执行applicationContext.getBean(beanName),congsingletonObject去匹配Bean实例返回

    # 2.属性填充/依赖注入

    通过populateBean()方法对Bean内部所需属性进行赋值填充

    注入的属性值会被BeanDefinition封装

    有如下三种情况 :

    • 普通属性 : String,int基本数据类型时,直接通过set方法的反射设置进去
    • 单项对象引用 : 如Service对象引用了Dao对象,从容器中getBean获取后通过set方法反射设置进去.如果容器中没有则先创建被注入对象Bean实例,再进行注入当前对象
    • 双向对象引用属性: 设计循环依赖问题,待会详讲 ......

    # 3.初始化

    初始化的三种方法及执行顺序:

    1. @PostConstruct方法注解
    2. InitializingBean接口
    3. @Bean(initiMethod="")注解

    如何进行初始化呢?

    通过**InitializeBean()**方法初始化Bean

    1. 通过invokeAwareMethods()方法为实现了各种Aware接口(信息感知接口)的Bean设置信息,如beanName,beanFactory等

    2. 通过invokeInitMethods()方法执行Bean的初始化方法

    3. 执行afterPropertiesSet()方法,即属性注入后执行的方法

    4. 执行自定义initMethod()方法

    5. 在第四步执行之前和之后,需要对Bean后处理器,BeanPostProcessors进行处理

    6. 注册销毁,通过**registerDisposableBean()**方法将实现类销毁接口DisposableBean的Bean进行注册

    常见后处理器

    • AnnotationAwareAspectJAutoProxyCreator : 负责AOP处理
    • 负责构造后**@PostConstruct和销毁前@PreDestroy**处理的lnitDestroyAnnotationBeanPostProcessor等
    • 实现BeanPostPressors接口的自定义后处理器

    # 4.销毁

    销毁Bean的三种方法及执行顺序:

    1. @PreDestroy方法注解
    2. 实现DispossableBean
    3. @Bean(destroyMethod="")

    执行postProcessBeforreDestruction()方法,销毁前处理器,也就是**@PreDestory**修饰的方法

    再执行destory()销毁

    最后通过invokeCustomDestoryMethod()方法去执行自定义destroyMethod方法

    # 问题详解

    # 循环依赖

    我们刚才说过,当注入属性时,两个对象相互依赖

    A内部需要注入B,B内部需要注入A

    有点像死锁 ......

    流程是这样的

    实例化Bean1 ---> 属性填充需要Bean2 ---> 判断容器中是否含有Bean2

    ---> 有,则初始化

    ---> 没有,则实例化Bean2

    如果从单例池中寻找肯定是找不到的

    所以先实例化Bean2,此时Bean2只经过了实例化

    没有走完生命周期,故还是个半成品

    Bean2也是如此,于是构成了死循环

    解决方案

    关键点在于再容器中判断是否够含有Bean2的这一步

    可以不从单例池中寻找

    转而提供半成品Bean2的Map中寻找,是Bean1维护者Bean的引用即可

    Spring实现引用的方案为 --- 三级缓存

    # 三级缓存

    Spring提供了三级缓存存储 完整Bean实例和半成品Bean实例,用于解决循环依赖问题

    在DefaultListableFactory的上四级父类DefaultSingletonBeanRegistry中提供如下三个Map:

    • 一级缓存 : 最终存储单例Bean承平的容器,即实例化和初始化都完成的Bean
    Map<String,0bject> singletonObjects = new ConcurrentHashMap(256);
    1
    • 二级缓存 : 早期Bean单例池,缓存半成品对象,且当前对象已经被其他对象引用了
    Map<String,0bject> earlySingletonobjects = new ConcurrentHashMap (16);
    1
    • 三级缓存 : 单例Bean的工厂池,缓存半成品对象,对象未被引用,使用在通过工程创建Bean
    Map<string,0bjectFactory<?>> singletonFactories = new HashMap(16);
    1

    所以循环依赖的解决流程如下 :

    1. 实例化Bean1,包一层ObjectFactory,放入三级缓存中
    2. 判断属性注入Bean2时,在三级缓存中找一遍,没找到
    3. 实例化Bean2,放入三级缓存
    4. 判断属性注入,找到了Bean1,将三级缓存中的Bean1放入二级缓存中
    5. 初始化Bean2完成,将Bean2由三级缓存放入一级缓存
    6. 继续初始化Bean1,完成后放入一级缓存

    通过**addSingleton()**方法将完整的对象放入单例池

    即可使用完整的Bean
