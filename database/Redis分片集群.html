    Redis分片集群

    lin 2023/11/21 Redis

    我们知道

    主从模式和哨兵模式可以分别解决高并发读和高可用的问题

    但是现在任然存在两个问题

    • 海量数据存储
    • 高并发写

    使用分片集群可以解决上述问题

    分片集群特征有 :

    • 集群中有多个master,每个master保存不同数据

    • 每个master都可以有多个slave节点,构成局部主从关系

    • master之间通过ping监测彼此健康状态,不需要哨兵机制

    • 客户端请求可以访问集群任意几点,最终都会被争取转发

    # 一.散列插槽

    查看集群信息时,我们发现每个master节点都有一部分slots

    Redis会把每一个master节点映射到0~16383共16384个插槽( hash slot)上

    每个节点都分配一部分

    我们知道,Redis是键值型数据库

    而数据key不是与节点绑定,而是与插槽绑定.

    redis会更具key的有效部份计算slots值 :

    • key中包含"{}",且"{}"中至少包含1个字符,则"{}"中的部分是有效部分
    • key中不包含"{}",整个key都是有效部分

    例如:

    key是num,num是有效部分

    key是{lin}num,则根据"lin"计算

    计算方式是利用CRC16算法得到一个hash值,然后对16384取余,得到的结果就是slot值

    就知道key应该存储到哪个节点了

    问题来了

    为什么key绑定插槽而不绑定节点?

    这是因为节点可以能出现宕机的情况

    绑定了插槽,在节点宕机后转移插槽或者扩容时转移插槽

    如何将同一类数据固定保存在同一个Redis实例?

    我们可以使key的有效部分一致,通过算法就获得了相同slot值,就能定位到同一个实例

    # 三.故障转移

    前面提到

    分片集群中没有哨兵模式

    但是也可以做到故障转移

    当集群中有一个master宕机会发生什么?

    1. 首先该实例与其他实例失去连接
    2. 然后疑似宕机
    3. 最后确定下线,自动提升一个slave为新的master

    这是意外宕机的自动故障转移

    但是Redis中的master节点可能会老旧需要维护

    我们需要做的就是手动故障转移

    在原的master节点利用cluster failover命令手动让集群中的某个master宕机,切换到命令执行的slave节点,实现无感知的数据迁移

    1. slave节点告诉master,节点拒绝任何客户端请求
    2. master返回当前数据offset给slave
    3. slave等待数据offset与master一致
    4. 开始故障转移
    5. 标记自己为master
    6. 开始处理客户端请求
