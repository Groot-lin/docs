    SQL成本分析

    lin 2023/12/26 SQL

    SQL调优是在日常生产中运用广泛

    近些年大厂面试越来越容易问

    一条SQL语句

    造成的影响还是挺大的

    那么在SQL调优之前

    MySQL查询优化器是怎么做的呢?

    接下来我们来了解一下

    # 一.成本

    MySQL执行一个查询可以有不同的执行方案,它会选择其中成本最低的方案去执行查询

    MySQL的成本主要由以下两部分组成:

    • I/O成本

      存储引擎一般将数据和索引存储到磁盘上,当我们进行查询时,需要将数据和索引加载到内存中,这段I/O所消耗的时间被称为I/O成本

    • CPU成本

      检测读取是否满足条件的记录,对结果及排序等等,被称为CPU成本

    对于InnoDB说,页是磁盘和内存之间交互的基本单位

    MySQL规定:

    读取一个页花费的成本是 --- 1.0

    读取检测一条记录的成本 --- 0.2

    1.0 或 0.2 都被称为 成本常数(可调节)

    # 二.单表查询的成本

    我们先准备一张表:

    CREATE TABLE single_table (
 id INT NOT NULL AUTO_INCREMENT,
 key1 VARCHAR(100),
 key2 INT,
 key3 VARCHAR(100),
 key_part1 VARCHAR(100),
 key_part2 VARCHAR(100),
 key_part3 VARCHAR(100),
 common_field VARCHAR(100),
 PRIMARY KEY (id),
 KEY idx_key1 (key1),
 UNIQUE KEY idx_key2 (key2),
 KEY idx_key3 (key3),
 KEY idx_key_part(key_part1, key_part2, key_part3)
) Engine=InnoDB CHARSET=utf8;

    # 基于成本的优化步骤

    在一条单表查询语句真正执行之前,MySQL 的查询优化器会找出执行该语句所有可能使用的方案

    对比之后找出成本最低的方案,这个成本最低的方案就是所谓的 执行计划

    之后才会调用存储引擎提供的接口真正的执行查询

    过程如下:

    1. 根据搜索条件，找出所有可能使用的索引
    2. 计算全表扫描的代价
    3. 计算使用不同索引执行查询的代价
    4. 对比各种执行方案的代价，找出成本最低的那一个

    我们用一个实例分析一下

    SELECT * FROM single_table WHERE 
 key1 IN ('a', 'b', 'c') AND 
 key2 > 10 AND key2 < 1000 AND 
 key3 > key2 AND 
 key_part1 LIKE '%hello%' AND
 common_field = '123';

    # 1.根据搜索条件,找出所有可能使用的索引

    我们逐个分析:

    • key1 IN ('a', 'b', 'c') ，这个搜索条件可以使用二级索引 idx_key1
    • key2 > 10 AND key2 < 1000 ，这个搜索条件可以使用二级索引 idx_key2
    • key3 > key2 ，这个搜索条件的索引列由于没有和常数比较，所以并不能使用到索引
    • key_part1 LIKE '%hello%',key_part1通过 LIKE 操作符和以通配符开头的字符串做比较，不可以适用索引
    • common_field = '123' ，由于该列上压根儿没有索引，所以不会用到索引

    所以可能使用到的索引就是idx_key1和idx_key2

    # 2.计算全表扫描的代价

    我们知道查询成本 = I/O成本+CPU成本

    所以我们需要两个信息:

    • 聚簇索引占用的页数
    • 该表表中的记录数

    MySQL为每个表维护了一系列的统计信息

    SQL语句如下:

    SHOW TABLE STATUS LIKE 'single_table'\G
    1

    然后我们会看到

    我们只关心两个数据:

    • Row --- 9693

      表中的记录数,在InnoDB中是一个估计值,

    • Data_length --- 1589248

      表占用的存储空间字节数对于InnoDB来说,该值相当于聚簇索引占用的存储空间大小

      Data_length = 聚簇索引的页面数量 * 每个页面的大小

      InnoDB每个页16kb

      聚簇索引的页面数量 = 1589248 /16 /1024 = 97

    现在可以计算全表扫描的代价了

    • I/O成本:

      97 * 1.0 + 1.1 = 98.1

      97表示页数,1.0表示加载一个页面的成本常数,1.1为一个微调值

    • CPU成本:

      9693 * 0.2 + 1.0 = 1939.6

      9693 指的是统计数据中表的记录数,0.2 指的是访问一条记录 所需的成本常数,后边的 1.0 是一个微调值

    • 总成本:

      98.1 + 1939.6 = 2037.7

    综上所述，对于 single_table 的全表扫描所需的总成本就是 2037.7

    # 3.计算不同索引执行查询的代价

    从第一步分析得知

    上述查询可能使用到idx_key1和idx_key2这两个索引

    MySQL 查询优化器先分析使用唯一二级索引的成本,再分析使用普通索引的成本,所以我们也先分析 idx_key2 的成本,然后再看使用 idx_key1 的成本

    # 1).分析key2

    key2的搜索条件为:

    key2 > 10 AND key2 < 1000
    1

    第一步,MySQL先从idx_key2索引中定位key2在(10,1000)这个区间的记录,找到这些记录对应的id列的值

    第二步,MySQL再从聚簇索引中更具上一步得到的id值找到完整用户记录

    对于二级索引 + 回表的方式,MySQL分析成本基于两个方面的数据:

    • 范围区间数量:

      无论区间占用多少页面,查询优化器认为读取索引的一个区间范围的I/O成本与读取一个页面是相同的,所以访问这个区间的二级索引成本为:

      1 * 1.0 = 1.0

    • 需要回表的记录数:

      先找到区间最左记录,再找到区间最右记录,这两个操作都是常数级别

      如果这两条记录相隔不远(MySQL5.7)是不大于10个页,则可以精确统计

      如果太远就向右读10个页,计算平均每个页中包含多少记录

    如何确定他们之间有多少记录呢?

    我们知道,索引的目录项对应一个数据页

    所以可以计算其父节点之间隔了几条记录

    根据上诉算法测得idx_key2在区间(10,1000)中包含大约95条记录

    读取95条二级索引的CPU成本就是

    95 * 0.2 + 0.01 = 19.01

    接着还需要做两件事:

    • 根据这些记录里的主键值到聚簇索引中做回表操作

      I/O成本为 95 * 1.0 = 95.0

    • 回表操作后得到的完整用户记录，然后再检测搜索条件是否成立

      这部分操作就是95 * 0.2 = 19.0

    所以idx_key2的查询成本为

    • I/O成本:

      95 * 1.0 = 96.0

    • CPU成本:

      95 * 0.2 + 0.01 + 95 * 0.2 = 38.01

    • 总成本:

      96.0 + 38.01 = 134.01

    综上所诉

    当计算二级索引+回表操作时

    需要分别计算二级索引的成本和回表的成本

    # 2).分析key1

    idx_key1 对应的搜索条件是：

    key1 IN ('a', 'b', 'c')
    1

    相当于三个单点区间

    第一步,先查询二级索引中这些记录对应的id列的值

    第二步,再从聚簇索引中根据上一步得到的一系列id值找到完整的用户记录

    我们直接进行计算

    • 范围区间数量:

      3 * 1.0 = 3.0

    • 需要回表的记录数

      (35 + 44 + 39) * 0.2 + 0.01 = 23.61

      即是CPU成本

    • 回表操作的成本

      118 x 1.0 = 118.0

    • 回表操作后得到的完整用户记录，然后再检测搜索条件是否成立

      118 x 0.2 = 23.6

    综上所述，使用 idx_key1 执行查询的总成本就是： 121.0 + 47.21 = 168.21

    # 4. 对比各种执行方案的代价，找出成本最低的那一个

    • 全表扫描的成本： 2037.7
    • 使用 idx_key2 的成本： 134.01
    • 使用 idx_key1 的成本： 168.21

    很显然,选择使用idx_key2来执行查询

    # 三.连接查询的成本

    连接查询至少有两张表吧

    我们再准备一张和之前那张表一样

    之前那张叫做s1,这张叫做s2吧

    # 1.连接的成本

    MySQL 中连接查询采用的是嵌套循环连接算法，驱动表会被访问一次，被驱动表可能会被访问多 次

    所以对于两表连接查询来说，它的查询成本由下边两个部分构成：

    • 单次查询驱动表的成本
    • 多次查询被驱动表的成本

    我们把对驱动表进行查询后得到的记录条数称之为驱动表的 扇出
