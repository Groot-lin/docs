    InnoDB页结构

    lin 2023/10/5 MySQL

    # 一.InnoDB页

    InnoDB是一个将表中的数据存储到磁盘上的存储引擎

    所以关机后存储数据依然存在.

    处理数据时 :

    • 读请求 : 将磁盘内容读到内存
    • 写请求 : 将内存中的被修改后的数据写回磁盘

    但是对于磁盘的读写操作非常慢,与内存操作相差了几个数量级.

    所以我们想要从表中获取数据时,InnoDB将数据划分为若干页,以页为磁盘和内存之间交互的基本单位,InnoDB中页的大小一般为16k

    在这种情况下,即使你select一条数据,但是数据所在页都会从磁盘加载到内存

    InnoDB分为数据页和索引页

    数据页用来存放数据

    索引页用来存放索引

    InnoDB的索引和数据存放在一起,这是和MYISAM不同的地方之一

    # 二.InnoDB页结构

    16k的页被划分成以下几个部分

    名称 中文名 作用 大小
    File Header 文件头部 页的一些通用信息 38B
    Page Header 页面头部 数据页专用的一些信息 56B
    Infimum + Supermum 最小记录和最大记录 两个虚拟的行记录,存放最大记录和最小记录 26B
    User Records 用户记录 实际存储的行数据内容，初始为空，随着记录的增加，从Free Space中挪用空间 不确定
    Free Space 空闲空间 页中未使用的空间，为User Records提供空间 不确定
    Page Directory 页面目录 页中的某些记录的相对位置 不确定
    File Trailer 文件尾部 校验页是否完整 8B

    # 1.User Records (真实数据)

    在页的7个组成部分中，我们自己的数据（student、user）会存储在User Records中。

    但是在一开始生成页的时候不会为User Redords分配空间，每当我们插入一条数据，都会从Free Space分出部分空间到User Redords来存储这条数据

    直到Free Space被User Records全部替代，说明这个页用光了，就会创建新的页。

    # 2. Page Director(页目录)

    记录之间类似链表,但是效率较慢.

    MySQL的索引使用的是B+树，这里我们讲的是数据页，不是索引页，所以没有用到B+树，但是MySQL的设计者还是采用了一定的方式加快数据的查询。

    MySQL采用了以下方式：

    1. 将所有未删除的数据划分为不同的组(包括最大记录和最小记录)
    2. 每个组的最后一条记录的记录头信息中的 n_owned 属性表示该组内有多少条记录
    3. 将每个组的最后一条记录的地址偏移量单独提取出来按照顺序存储到 Page Directory 中（也就是页目录），我们将这些地址偏移量称为 槽(Slot)

    所以页目录由槽组成

    # 3. Page Header(页头部信息)

    MySQL为了能得到一个数据页中存储记录的状态信息

    专门开辟了一个叫Page Header的部分

    名称 作用
    Page_N_Dir_Slots 该页的页目录中的槽的数量
    Page_N_Heap 本页中的记录的数量（删除&未删除）
    Page_Level 当前页在B+树所处的层级
    Page_Index_id 本页属于哪个索引

    # 4. File Header（文件头）

    File Header是每种页都有的属性，其中重要的字段如：本页编号、上一页编号、下一页编号、校验和。

    上一个与下一个页的编号：可以将所有数据页组成一个双链表，遍历就很方便了。

    下一个数据页中用户记录的主键值必须大于上一个页中用户记录的主键值

    # 5.File Trailer（文件尾）

    判断页是否完整

    Compact行格式 B+树索引
